# Demo 设计文档：多 Agent 共识 · 健康第二意见

> 代号（暂定）：**Council** —— "一场你看得见的多专家会诊"
> 定位：对标蚂蚁阿福，但走差异化——**不给一个答案，而是把多个专家级 Agent 的辩论、分歧、理由和共识透明呈现**
> 目标：① 对上交付一个"医疗 + 多 Agent"产品 ② 实测验证共识引擎 ③ 拿去探真实市场反馈

---

## 1. 一句话产品定义

> 用户提出一个复杂健康/医疗问题 → 多个**异构、独立**的专家 Agent 各自给出意见 →
> 它们互相辩论、给论据、多轮磋商 → 系统**透明呈现**：谁怎么看、在哪分歧、各自理由、最终共识 / 未达成共识的地方。
> **我们卖的不是"答案"，是"看得见的会诊过程"和"被诚实标定的不确定性"。**

---

## 2. 为什么这个形态（差异化逻辑）

| 维度 | 蚂蚁阿福 | 本 Demo（Council） |
|---|---|---|
| 核心价值 | 快、免费、给一个健康建议 | 复杂问题的多视角权衡 + 分歧透明 |
| 形态 | 单一 AI 健康朋友 | 可见的多专家会诊 |
| 适合问题 | 日常、常见、有标准答案 | 复杂、有分歧、需权衡 |
| 壁垒 | 流量+医药险生态 | 异构多 Agent 共识引擎 |
| 监管 | 绕开诊断（咨询） | **同样绕开诊断**：只呈现推理/选项，不下诊断结论 |

**关键设计原则**：阿福做的是"健康问题的最短路径"；我们做的是"复杂问题的最透明路径"。两者不正面竞争。

---

## 3. 核心体验（UX 流）

1. **提问**：用户输入复杂问题（症状组合 / 体检异常解读 / 两种治疗方案怎么选 / 多病共存的取舍）。
2. **组建会诊团**：系统按问题领域召集 N 个专家 Agent（如：内分泌、心内、循证医学审查员、患者沟通代表）。**界面展示每个 Agent 的身份、知识库来源、思维框架（Skill）。**
3. **第一轮独立意见**：每个 Agent 独立给观点 + 理由 + 引用证据。**并排展示，让用户看到原始分歧。**
4. **多轮辩论**：Agent 互相质疑、补充、修正，界面以"会诊记录"形式滚动呈现关键交锋。
5. **共识收敛**：系统输出
   - ✅ **达成共识的部分**（带支撑理由与证据）
   - ⚠️ **未达成共识 / 高不确定性的部分**（明确标出，建议线下就医）
   - 📋 **结构化选项**（每个选项的利弊、适用条件）
6. **安全出口**：全程显著免责声明——"本服务提供信息参考，不构成诊断，请以执业医师面诊为准"；高风险信号触发"建议立即就医"硬提示。

> **设计灵魂**：分歧不是 bug，是 feature。把"专家都拿不准的地方"诚实地标给用户，正是单模型和免费咨询给不了的价值。

---

## 4. 引擎设计（与压力测试咬合）

| 设计点 | 做法 | 对应验证 |
|---|---|---|
| **异构底座** | 不同厂商/尺寸的基座模型扮演不同 Agent（非同一模型换皮） | 实测"错误重合度"应显著低 |
| **独立知识库** | 每个 Agent 挂独立检索库（指南/文献/科室知识） | grounding 兜底，防集体幻觉 |
| **独立 Skill（思维框架）** | 每个 Agent 有自己的推理框架（如循证审查员专做证据质量评级） | 保证分歧来自真实视角差异 |
| **外部裁判 / grounding** | 关键结论必须可回溯到检索证据；无证据则标"低置信" | 防"靠说服力收敛"而非"靠正确性" |
| **收敛准则** | 论据质量驱动，而非多数票；保留少数派意见 | 防 sycophancy 假性收敛 |

**内嵌的引擎压力测试（与 Demo 同步跑）**：
- **测试 A 价值**：硬基准（如 MedQA）上，Council vs 单模型 vs 单模型+self-consistency 的准确率。关键是 **Council 是否赢过 self-consistency**。
- **测试 B 独立性**：N 个 Agent 错误重合度。
- **测试 C 抗说服**：注入"似是而非"的错误论据，看是否被带跑。
- **测试 D 分歧信号**：未收敛 case 是否恰好是"真正困难、该转人工"的 case。

> 这四个测试的结果直接决定产品叙事能不能成立、以及该不该把引擎推向更高风险场景。

## 5. 技术架构（Demo 级）

```
[Web 前端：会诊室 UI]
   │  问题输入 / 会诊记录可视化 / 共识与分歧展示
[编排层：共识引擎 Orchestrator]
   │  Agent 召集 → 多轮辩论调度 → 收敛判定 → 不确定性标定
[Agent 层：N 个异构专家 Agent]
   │  每个 = 基座模型 + 独立知识库(RAG) + Skill 框架
[证据层：检索/grounding]
   │  临床指南库 / 文献库 / 结构化知识（外部合作提供）
[安全层]
   └  免责声明 / 高风险信号拦截 / 不给诊断结论的输出约束
```

- **不接医院 HIS/PACS、不私有化部署**（Demo 阶段避开数据合规与采购）。
- **不调真实患者数据**，用公开病例/合成案例演示。

---

## 6. Demo 范围（明确 In / Out）

**In（两周做到）**
- 1–2 个垂直场景（建议：体检报告异常解读 / 两种方案怎么选——分歧天然明显）。
- 3–4 个异构专家 Agent + 1 个循证审查 Agent。
- 完整的"独立意见 → 辩论 → 共识/分歧"可视化。
- 内嵌四项压力测试的离线评测脚本与结果。

**Out（明确不做）**
- 不下诊断结论、不开处方、不做影像读片（避监管）。
- 不接医院系统、不收患者费用（避合规/采购）。
- 不追求覆盖所有科室（先证明机制，不证明覆盖度）。

## 8. 这个 Demo 同时回答了三个问题

1. **对上**："医疗 + 多 Agent" 的产品有了，可演示。
2. **对内**：引擎到底是护城河还是噱头，有数据说话。
3. **对外**：拿差异化形态去探医生端/高需求用户端反馈，**用市场反推商业策略**。

> 核心提醒：本 Demo 的杀伤力来自**"展示分歧比给答案更有价值"**这个反直觉的卖点。守住"不给诊断结论"的边界，它就既安全又独特。
